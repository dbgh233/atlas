<objective>
Build the opportunity matching engine that connects a Calendly webhook event to the correct GHL opportunity -- using Calendly Event ID as primary match and email+type+stage as fallback.

Purpose: This is the core matching problem. A webhook fires with an event UUID and invitee email. Atlas must find the exact GHL opportunity that corresponds to this Calendly event. Getting this wrong means writing field updates to the wrong deal.
Output: A matcher module that takes a WebhookEvent and returns a MatchResult with the matched opportunity (or None with reason).
</objective>

Create `app/modules/webhooks/matcher.py`:

Dataclass `MatchResult`:
- `opportunity: dict | None` — the matched GHL opportunity (full object), or None
- `opportunity_id: str | None` — the GHL opportunity ID
- `match_method: str` — "calendly_event_id", "email_fallback", or "none"
- `match_reason: str` — human-readable explanation of what happened
- `appointment_type: str | None` — the resolved appointment type (from GHL opp, which may differ from Calendly)

Async function `match_opportunity(ghl_client: GHLClient, event: WebhookEvent) -> MatchResult`:

**Step 1: Primary match — Calendly Event ID**
- Call `ghl_client.search_opportunities()` to get all open opportunities in the pipeline
- Iterate through opportunities and check each one's customFields for the Calendly Event ID field (ID: `U3dnzBS8MNAh8Gl6oj07`)
- Custom fields on GHL opportunities are in `opp.get("customFields", [])` — each is `{"id": "...", "value": "..."}` OR `opp.get("customFields", {})` as a dict. Handle both formats.
- The Calendly Event ID field stores the full scheduled event URI (e.g., "https://api.calendly.com/scheduled_events/UUID")
- Match if the stored URI equals `event.scheduled_event_uri` OR if the stored value contains `event.calendly_event_uuid`
- If found: extract the opportunity's Appointment Type from custom field `g92GpfXFMxW9HmYbGIt0`
- Return MatchResult with match_method="calendly_event_id"
- Log: "match_primary_success" with opp_id and opp_name

**Step 2: Fallback match — email + Appointment Type + stage**
- Only reached if primary match fails
- Log: "match_primary_miss", attempting fallback
- Call `ghl_client.search_contacts(event.invitee_email)` to find the contact
- If no contact found: return MatchResult(opportunity=None, match_method="none", match_reason="No GHL contact found for email {email}")
- Get the contact's ID and search for opportunities associated with this contact
  - The search_opportunities result includes `contact.id` on each opportunity — filter the already-fetched opportunities by `opp.get("contact", {}).get("id") == contact_id`
  - Alternatively, iterate the full opp list and check `opp.get("contactId")` or `opp.get("contact", {}).get("id")`
- Filter candidates by relevant stage:
  - For Discovery events: opportunity must be in Discovery stage (ID: `16634e86-5f37-4bda-85a0-336ad5c744d8`)
  - For Onboarding events: opportunity must be in Onboarding Scheduled stage (ID: `96f0eb52-c557-45c8-b467-d2cce611ffb2`)
- Filter candidates by Appointment Type custom field (g92GpfXFMxW9HmYbGIt0):
  - For Discovery events: Appointment Type should be "Discovery"
  - For Onboarding events: Appointment Type should be "Onboarding"
- If exactly one candidate remains: match found
- If multiple candidates: pick the most recently created one (by `opp.get("createdAt")`) and log a warning "match_fallback_ambiguous"
- If zero candidates: return no match

**Step 3: Appointment Type resolution (EVNT-06)**
- When a match is found (either method), extract the GHL opportunity's Appointment Type
- The GHL opportunity's Appointment Type is the source of truth — even if the Calendly event name says "Discovery" but GHL says "Onboarding", trust GHL
- Store the resolved appointment_type on MatchResult

  - Dict format: `{"xxx": "yyy", ...}`
- Handle both. Return the value as string or None if not found.
